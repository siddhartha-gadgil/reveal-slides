  • AI systems have shown impressive capabilities in Mathematical reasoning, including a Gold medal score at the IMO, and have become genuinely helpful in mathematical research.
  • However, they still have (related) limitations:
    • Accuracy: in mathematics errors cascade.
    • Autonomy: as they need human verification.
    • Scalability: limited by needing human guidance.
  • An approach is to use a Formal Proof System for verification, and Autoformalization as a bridge.
  • We specifically use LeanLang as our formal proof system.

Lean Prover

  • Lean is a functional programming language with a fast runtime which is self-hosting.
  • The type system of Lean is based on the Calculus of Inductive Constructions (CIC), which is a dependent type theory.
  • This allows arbitrary mathematical statements to be expressed as types.
  • Theorems and proofs thus are essentially definitions, with types representing statements and values representing proofs.

Lean Prover

  • Formalization in Lean thus allows for programs, algorithms, and proofs to be written in a unified framework.
  • We get not only guaranteed correctness but also modularity with enforced abstraction boundaries.
  • Lean is mostly written in Lean and has a pleasant meta-programming interface.
  • Thus, Lean provides a common language that is mathematically and Turing complete, is easily extensible, and with excellent tooling and engineering (e.g., FBIP, IDE support).

LeanAide:

Combinatory Autoformalization with Agentic AI

AutoFormalization: Ingredients and Stages

  • Formalize statements and definitions using an LLM given examples with results filtered (local step).
  • For documents/proofs (global steps):
    • Generate/Rewrite in a formalization-friendly style.
    • Generate a structured JSON representation of the mathematical text according to a formal schema.
    • Translate iteratively into Lean code, using:
      • Translation of statements and definitions.
      • Lean meta-programming to build Lean code from Lean theorems and definitions.
      • Automation to complete proofs.

Local Steps: Prompting & Post-processing

  • Suppose we are given the statement of a theorem.
  • We find (say) 20 documentation strings in Lean's mathematical library Mathlib that are closest to the statement (for OpenAI embeddings) and include them in the prompt.
  • We query GPT-4 with the prompt for (say) 10 responses.
  • We discard responses that are not valid Lean code (do not parse or do not elaborate).

Global Steps: Document to Json

  • We start with a document containing the theorem statement and proof, or just with a statement and get AI to generate the proof.
  • The proof is written/rewritten in a formalization-friendly style.
  • The document is scrutinized by an LLM/agent (like Codex) based on detailed instructions about common mathematical errors and stress-tests.
  • An agentic framework (itself vibe-coded) then generates JSON iteratively based on detailed schemas.

Combinatory Code Generation

  • Code generation is done recursively putting together code generation functions that take in JSON and output Syntax.
  • A function getCode combines these.
    • 						partial def getCode  (translator: CodeGenerator) (goal? : Option MVarId) 
    (kind: SyntaxNodeKinds) (source: Json) : 
			TranslateM (Option (TSyntax kind)) :=
  • The code generation functions can call getCode for recursive code generation.

theorem Nat.even_or_even_succ : ∀ (n : ℕ), Even n ∨ Even (n + (1 : ℕ)) :=
    by
    intro n
    induction n with
    | zero => simp only [Even.zero, zero_add, Nat.not_even_one, or_false]
    | succ n
      ih =>
      if condition_16468389163427038829 : Even n then grind only [= Nat.even_iff]
      else if condition_3270161773976375427 : Even (n + (1 : ℕ)) then grind only else grind only
      done
    done

LeanAide: Advantages

  • There are many powerful Autoformalization systems such as Aristotle, Gauss and AxiomProver.
  • Coding agents and LLMs have also become good at generating Lean code.
  • However, the nature of LeanAide makes it extensible and customizable. One also hopes that it can be scaled to larger proofs and that the generated Lean code is of good quality.
